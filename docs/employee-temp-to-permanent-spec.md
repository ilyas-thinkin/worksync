# Spec: Temporary → Permanent Employee Lifecycle

**Status:** Implemented (2026-07-01) — see checklist §12
**Owner:** IE / Admin panel
**Author decisions captured:** 2026-07-01
**Applies to:** WorkSync MES — `backend/src/routes/api.routes.js`, `backend/src/utils/qr.js`, `backend/src/migrations/`

> This document is the single source of truth for implementing the temp→permanent
> employee lifecycle. Any model/engineer implementing this MUST follow it exactly.
> Do not invent alternative approaches. If reality contradicts this spec, stop and
> report rather than improvising.

---

## 1. Problem statement

When an employee joins, HR issues a **temporary ID** and we generate a QR badge for
them. For a trial period (days/months) we track their performance. If they perform
well, they are converted to **permanent** and HR issues a **brand-new permanent code**.

Requirement: **all data collected during the temporary period (efficiency, hourly
output, workstation assignments, attendance, OT, defects, history) must remain
attached to the same employee after conversion — no data loss, no fragmentation.**

Some employees are hired **directly as permanent** and never have a temp phase.

---

## 2. Core design principle (READ THIS FIRST)

Separate **identity** from **printed code**:

| Concept | Column | Purpose | Mutable? |
|---|---|---|---|
| DB identity | `employees.id` (int PK) | Internal FK target. All history hangs off this. | **NEVER** |
| QR identity | `employees.uuid` (UUID) | What's encoded in the QR's scannable payload for employees created after this feature. Auto-generated by the DB (`DEFAULT gen_random_uuid()`) on every insert — no app code needed. | **NEVER** |
| Business code | `employees.emp_code` | Human-readable code printed on the badge (temp → permanent) | Yes, at conversion |

**Why this works:** All 15 tables that reference an employee FK to `employees.id`
(the integer PK) — NOT to `emp_code` and NOT to the QR. Therefore, if we keep the
**same row** (same `id`) for the employee's whole life and only change `emp_code`,
every historical record stays attached automatically with zero migration.

`uuid` exists purely so the QR's scannable content never has to reference the mutable
`emp_code` — it is a second, QR-facing identity, generated once at creation and never
touched again, even across temp→permanent conversion.

**Golden rule:** Temp→permanent conversion is an **UPDATE of the existing row**.
NEVER create a second employee record for the same person.

---

## 3. How the QR works after this change

- Employees created after this feature get a QR payload of
  `{ type:'employee', uuid, name }` — **`uuid` replaces `id` as the scannable
  identity field.** `uuid` is auto-generated by the DB at insert time and never changes.
  `emp_code` is intentionally NOT embedded in the payload — it's mutable (changes at
  conversion) and resolution never needs it since `uuid` always resolves first;
  embedding it would only bake a value that can go stale into permanent scannable
  data. The printed label (visible text under the QR) still shows the current code —
  that's built from live DB fields, not from this payload, so nothing is lost visually.
- Employees that existed before this feature keep their original payload,
  `{ type:'employee', id, code, name }` — **never touched, never regenerated.**
- The QR's **scannable payload/matrix graphic is NEVER regenerated on conversion**,
  for anyone. The physical QR pattern issued at hire time is kept for life, byte-identical.
- **The printed code line (the small text below the name) IS updated on conversion**
  — in place, editing only that `<text>` element in the existing SVG file. The name
  line and the QR matrix are left untouched. This is not a regeneration: same file,
  same scannable graphic, just the visible code text refreshed so a reprint shows the
  new permanent code under the same badge.
- Implementation: `qrUtils.updateEmployeeQrCodeLabel(id, newCode)` in
  `backend/src/utils/qr.js` — reads the existing SVG, regex-replaces only the
  `font-weight="normal"` text element (the code line), and writes it back. It does
  NOT call `generateEmployeeQrById` (which would re-derive the QR matrix from a new
  JSON payload and change the scannable graphic).
- Scanning always resolves correctly regardless of what text is printed, because the
  scan resolver reads `uuid` (or, for legacy QRs, `id`) from the payload (§6), not the
  printed text.

Current QR facts:
- QR file: `qrcodes/employees/{id}.svg` — named by the internal `id` (`qr.js:101`),
  this file-naming scheme is unchanged; only the JSON payload inside changed.
- New payload: `{ type:'employee', uuid, name }` (`qr.js:104-108`) — no `code` field.
- Legacy payload (pre-existing employees, untouched): `{ type:'employee', id, code, name }`.
- Generated by `generateEmployeeQrById(id)` in `backend/src/utils/qr.js` — called only
  at employee creation (and on manual edit via the admin PUT route, which is
  pre-existing behavior outside this feature's scope).

## 4. Scope (HARD CONSTRAINT)

- **Existing employees' QRs must NOT change in any way.** Do NOT regenerate, reprint,
  rename, or modify any existing employee QR file. Do NOT change any existing
  employee's `emp_code`. Do NOT backfill or convert them. They are already working and
  must be left exactly as-is.
- **New employees only** get the temp→permanent lifecycle fields and flow.
- The scan-resolver change (§6) is **backend-only and additive** — it does NOT alter,
  regenerate, or reprint any physical QR. It reads the `id` that is already present in
  every existing QR payload and falls back to the exact current `emp_code` lookup, so
  existing employees resolve to the same person with zero migration and no reprint.
- If any step in implementation would require regenerating or reprinting an existing
  employee's QR, STOP — that is out of scope and not allowed.

## 5. Schema changes (migration)

Create the next sequential migration file in `backend/src/migrations/`
(check the latest applied number first; per project memory it was 033).

```sql
-- Employee lifecycle: temporary -> permanent

ALTER TABLE employees
  ADD COLUMN employment_status VARCHAR(20) NOT NULL DEFAULT 'permanent'
    CHECK (employment_status IN ('temporary','permanent')),
  ADD COLUMN permanent_from DATE;

-- Default is 'permanent' so existing rows are unaffected / treated as already-permanent.
-- New employees created via the temp flow will be inserted with 'temporary'.

CREATE TABLE employee_code_history (
  id            SERIAL PRIMARY KEY,
  employee_id   INTEGER NOT NULL REFERENCES employees(id) ON DELETE CASCADE,
  old_emp_code  VARCHAR(50) NOT NULL,
  new_emp_code  VARCHAR(50) NOT NULL,
  changed_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  changed_by    INTEGER
);

CREATE INDEX idx_employee_code_history_employee ON employee_code_history(employee_id);
```

**Constraint reminder:** `employees.emp_code` already has
`CHECK (emp_code ~ '^[A-Z0-9]+$')` — uppercase letters + digits only, no dashes or
symbols. The new permanent code entered by HR MUST satisfy this. `emp_code` stays
UNIQUE; the freed temp code becomes reusable (safe because we resolve by uuid/id — §6).

A second migration adds the `uuid` identity column:

```sql
ALTER TABLE employees
  ADD COLUMN uuid UUID NOT NULL DEFAULT gen_random_uuid();

ALTER TABLE employees
  ADD CONSTRAINT employees_uuid_key UNIQUE (uuid);
```

`gen_random_uuid()` is built into PostgreSQL core since v13 — no extension required
(confirmed on this project's PG 17). Adding this column backfills a UUID for every
existing employee too (schema consistency), but since existing QR files are never
regenerated, this has zero effect on any existing badge.

## 6. Scan-resolver change (CRITICAL — this is what makes it work)

Every employee-QR scan resolver goes through one shared helper,
`resolveEmployeeFromQrPayload(parsed)`, defined next to `parseSupervisorQr` in
`backend/src/routes/api.routes.js`. It tries, in order:

1. `parsed.uuid` (validated against a UUID regex first, to avoid a Postgres type
   error on garbage input) → `SELECT ... WHERE uuid = $1 AND is_active = true`
2. `parsed.id` → `SELECT ... WHERE id = $1 AND is_active = true`
3. `parsed.code || parsed.emp_code || parsed.raw` → `SELECT ... WHERE emp_code = $1 AND is_active = true`

Returns the first match, or `null`. All employee-QR-consuming routes call this
instead of duplicating the fallback chain:
- `POST /supervisor/resolve-employee-by-qr`
- `POST /supervisor/resolve-employee`
- `POST /supervisor/assign`
- `POST /supervisor/scan` (requires `uuid` or `id` present before attempting resolution;
  `code`-only payloads are rejected here specifically, matching this route's original
  strictness)

This is what makes both the uuid-based (new employees) and id-based (existing
employees) QR payloads resolve correctly through the same code path, and it's what
keeps every previously printed badge — old or new — scanning correctly forever
regardless of how many times `emp_code` changes on the underlying record.

**This is a backend-only change. It does NOT regenerate, reprint, or modify any
physical QR — existing or new.** It only changes how a scanned payload is looked up.
Existing employees' QRs already carry `id`, so they resolve identically; nothing on
their side changes.

## 7. Create-employee flow (new hires)

Extend the existing employee-create endpoint (around `api.routes.js:5474`
`INSERT INTO employees ...`) to accept an `employment_status`:

- **Temporary hire:** insert with `employment_status='temporary'`, `permanent_from=NULL`,
  and the HR-issued temporary `emp_code`. Generate QR via `generateEmployeeQrById(id)`
  after insert (unchanged behavior).
- **Direct-permanent hire:** insert with `employment_status='permanent'`,
  `permanent_from=CURRENT_DATE`, and the permanent `emp_code`. Same QR generation.

No separate code path is needed beyond passing the status + permanent_from.

---

## 8. Conversion flow — `POST /employees/:id/make-permanent`

New endpoint. Body: `{ new_emp_code }` (HR-entered, brand-new permanent code).
Run everything in **one transaction on the same row**:

1. Load current employee by `:id`. If not found or already `permanent`, return 400.
2. Validate `new_emp_code`:
   - matches `^[A-Z0-9]+$`
   - not already used by another employee (UNIQUE)
3. `INSERT INTO employee_code_history (employee_id, old_emp_code, new_emp_code, changed_by)`
   capturing the old temp code.
4. `UPDATE employees
      SET emp_code = $new,
          employment_status = 'permanent',
          permanent_from = CURRENT_DATE,
          updated_at = NOW(), updated_by = $user
    WHERE id = $id`
5. Call `qrUtils.updateEmployeeQrCodeLabel(id, new_emp_code)` — updates ONLY the
   printed code line in the existing `qrcodes/employees/{id}.svg`. The QR matrix
   (scannable graphic) and name line stay byte-identical; this is not a regeneration.
6. Write an `audit_logs` entry.
7. `realtime.broadcast('data_change', { entity: 'employees' })` (or existing pattern)
   so IE/admin/supervisor UIs refresh.

Because `id` never changed and all history FKs to `id`, **all temp-period data remains
attached with no migration.**

## 9. UI (IE / Admin panel)

- Show `employment_status` badge (Temporary / Permanent) on the employee list/row.
- On a temporary employee, add a **"Make Permanent"** action that opens a small form:
  - input for the new permanent code (validated `^[A-Z0-9]+$`, uniqueness checked)
  - on submit → `POST /employees/:id/make-permanent`
  - on success → toast confirming the code changed; no QR reprint is offered or needed.
- Employee create form: add a Temporary/Permanent selector for new hires.
- Frontend files: `backend/src/public/js/ie.js` and/or `admin.js` per where employee
  management lives.

## 10. Reports (merged, optionally splittable)

- Default: reports stay continuous per employee (already true — history hangs off `id`).
- Enhancement (optional, can be a later phase): allow splitting an employee's metrics
  at `permanent_from` to compare **trial-period** vs **permanent-period** performance.
  This is valuable because trial performance is what justifies the promotion.

---

## 11. Explicit non-goals / accepted trade-offs

- We do NOT create a second employee record on conversion. One row for life.
- We do NOT migrate, reprint, regenerate, or otherwise touch existing employees or
  their QR badges. They stay exactly as they are today.
- On conversion, the QR **matrix graphic is never regenerated** for anyone — only the
  printed code line (text) is updated in place on the employee's own existing SVG.
- A freed temp `emp_code` may be reused by a future hire — safe, because scanning
  resolves by `id`, not code.
